既存レセコンの「夜間に全件を止めて一括処理する」設計を yrese に持ち込まないことを確定する。
24/365 稼働(ARC-010)の前提であり、本方針の緩和はアーキテクチャ全体の緩和を意味する。

## 1. 禁止する設計

次のいずれかに該当する設計・実装は、リスクレベルにかかわらず導入禁止とする。

1. **業務停止を前提とする一括処理**: 処理中に受付・調剤・会計のいずれかを止める(またはロックで実質止まる)夜間・定時の全件処理。
2. **「翌日にならないと正しくならない」データ**: 日次バッチが走るまで残高・状態・集計が不正確なまま業務画面に出る設計。
3. **時刻依存の暗黙処理**: 特定時刻の到来自体をトリガーに、明示的な業務操作なしでデータの確定・締め・状態遷移を行う仕組み(暗黙の現在時刻禁止の規律と同根)。

## 2. 置換原則

| 従来のバッチ用途 | yrese での置換 |
|---|---|
| 日次の集計・台帳更新 | 逐次イベント処理(Outbox/Inbox)+オンデマンド/逐次投影。投影は再計算可能で、破棄・再構築しても正本(イベントストア)を損なわない |
| 夜間の外部照会まとめ送信 | Official Adapter 経由の逐次・随時処理。外部側の受付時間制約は PENDING 系ステータスで表現し、業務は止めない |
| 月次の請求データ一括生成 | 明示的な業務操作としての月次締め(§3) |
| 深夜のマスター更新 | 有効日付き(effectiveFrom)のバージョン切替。事前投入し、適用日到来で新版が参照される(データ停止を伴わない) |

- 重い処理は「業務を止めて流す」のではなく、業務と並行して流せる粒度に分割する。
- 投影の遅延・失敗は観測対象(OPS-005)とし、失敗時は投影のみ再実行する(正本に触れない)。

## 3. 月次請求締めの扱い

- 月次締め・請求前点検・レセプト確定は**夜間バッチではなく、権限を持つ利用者による明示的な業務操作**である。
- 実行可否は `allowsClaimFinalization`(NORMAL モードのみ)と deny-by-default 権限に従う。時刻をトリガーに自動実行しない。
- 締め処理の実行中も、他業務(受付・調剤・会計)を止めない設計とする。
- 締め結果は監査対象の業務イベントとして記録する(誰が・いつ・どの範囲を締めたか)。

## 4. 例外手続

バッチ的処理(定時の全件処理)をどうしても導入する場合は例外であり、次を全て満たすこと。

1. fable5 起案の SSOT(本書の改版または個別 SSOT)で、対象・理由・業務影響・代替不能性を明文化する
2. opus4.8 レビュー+人間レビューの承認を得る(承認前の実装は BLOCKED_NOT_READY)
3. 業務を止めないこと(止まる場合は ARC-010 §4 の全停止例外手続を併用)
4. 失敗時の再実行が冪等であること

## 5. 停止条件(fail-closed)

- §1 に該当する設計の実装着手 → SSOT_UPDATE_REQUIRED(本書の例外手続なしには進めない)
- 時刻トリガーによる暗黙のデータ確定 → 実装禁止
- 投影失敗を無視して業務データを見せ続ける実装 → 導入禁止(遅延・失敗は明示表示と観測の対象)
