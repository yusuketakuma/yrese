## 1. 目的と結論

イベントソーシング(ES)を「どこに適用し、どこに適用しないか」の判断基準を確定する。

**結論: ES は全面採用しない。「法令・監査・請求の説明責任のために履歴そのものが正本であるべき集約」に限定して適用し、それ以外は通常の状態永続化とする。**

## 2. 適用判断基準

次の全てを満たす集約に ES を適用する。

1. 過去の任意時点の状態・判断根拠を**再現して説明する義務**がある(監査・返戻対応・遡及訂正)
2. 訂正が「上書き」ではなく「新しい業務イベント」として表現されるべきである(silent fix 禁止)
3. 状態遷移が DOM-004 で定義された有限の業務イベントに対応する

| 対象(適用) | 根拠 |
|---|---|
| 算定・請求ライフサイクル(算定実行・確定・返戻・再請求) | ARC-007 の immutability(確定 append-only)と DOM-004 の業務イベント対応が主根拠(QUA-009 の fail-closed 集計は副次的な受益) |
| 会計台帳(入金・未収金・一部入金・返金・調整) | ACC 系 SSOT の append-only 原則と同一 |
| 監査イベント | MOD-008(既に append-only。ハッシュ連鎖は SEC-007 改版で確定) |

| 対象外(適用しない) | 扱い |
|---|---|
| マスターデータ(医薬品・薬価等) | 版管理された参照データ(MST-001)。ES ではなく版スナップショット |
| 患者・保険の基本属性 | 通常永続化+変更監査イベント(履歴が正本である必要まではない) |
| UI 状態・画面下書き | 永続化対象外または一時保存 |

適用可否が曖昧な新集約は、本書の改版(PRC-007)で確定するまで ES を適用しない(既定は非適用 = 追加の複雑さを既定にしない)。

## 3. @yrese/events エンベロープとの関係(混同禁止)

- **@yrese/events(MOD-006)は同期イベント(Outbox/Inbox)のエンベロープ**であり、拠点間・システム間の伝搬が目的。**ES のイベントストア(ドメインイベントの正本保存)とは役割が異なる。**
- ドメインイベントを他拠点・他システムへ運ぶ場合は、イベントストアのレコードをエンベロープに**包んで**転送する(エンベロープ形式をイベントストアのスキーマとして流用しない)。
- **PHI 不変条件の継承**: ドメインイベントのペイロードが PHI を含む場合(調剤・患者関連はほぼ常に該当)、エンベロープ転送時は phiClassification≠none → encryptionStatus 'encrypted' が実行時強制される(MOD-006)。イベントストア側の保存時暗号化要件は WP-0047(WORM/テナント分離)で確定する。
- dead_letter には deadLetterReason 必須(MOD-006 実装済み不変条件)。ES 合流時の変換不能イベントも同じ規律で PENDING に隔離し、無断変換・無断破棄しない。

## 4. イベントスキーマの版管理

- イベント型は追記のみ(後方互換の破壊禁止)。意味が変わる場合は新イベント型を発行し、旧型は読み取り専用として残す。
- リプレイは CAL-010 と同じ決定論規律に従う(リプレイ時に現在時刻・現行マスターを暗黙参照しない。イベントに当時の版参照を記録する — CAL-009)。

## 5. 永続化技術(未確定)

候補として追記型に適する構成(例: RDB の append-only テーブル、S3 への不変オブジェクト)を検討対象とするが、**製品・構成の確定は BLOCKED**(要件: WP-0045 の 24/365・no-nightly-batch、WP-0047 の WORM・テナント分離と同時に満たすこと)。確定前にストレージ依存の実装を書かない。

## 6. 停止条件(fail-closed)

- 適用基準を満たさない集約への ES 適用(または基準を満たす集約の通常永続化) → SSOT_UPDATE_REQUIRED
- エンベロープ形式とイベントストアスキーマの混用 → CHANGES_REQUESTED
- PHI を含むイベントの平文転送・平文ログ → 実装禁止(MOD-006 実行時例外)
- 永続化技術の先行確定・依存実装 → BLOCKED
