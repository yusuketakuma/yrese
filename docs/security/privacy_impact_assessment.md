## 1. 取り扱う個人情報の類型

| 類型 | 例 | 分類 | 保存場所 |
|---|---|---|---|
| 患者基本情報 | 氏名・カナ・生年月日・性別・連絡先 | 個人情報 | Cloud Core + Edge(複製) |
| 保険・公費情報 | 保険者番号・記号番号・公費受給者番号・負担割合 | 個人情報(識別性高) | 同上 |
| 調剤・処方情報 | 処方内容・調剤記録・疑義照会・服薬指導記録 | **要配慮個人情報**(健康情報) | 同上 |
| 資格確認結果 | 資格有効性・限度額情報・(同意時)薬剤情報等 | 要配慮個人情報を含む | スナップショット最小化 |
| 請求情報 | レセプトデータ・返戻・再請求 | 要配慮個人情報を含む | Cloud Core(請求月ロック後は不変) |
| 職員情報 | 薬剤師・事務員のアカウント・操作履歴 | 個人情報 | Cloud Core |

## 2. 処理目的(目的外利用の禁止)

1. 調剤業務・薬学管理の実施(調剤録・薬歴連携)
2. 調剤報酬の算定・請求(レセプト作成・オンライン請求受け渡し)
3. 法令上の記録保存義務の履行
4. 業務継続(オフライン複製・バックアップ)
5. 監査・医療安全(操作証跡・請求証跡)

**禁止**: 広告・マーケティング目的の患者情報利用、患者誘導(v0.1.7 §7)、本番データでのテスト(§10/§36)、訓練環境での本番個人情報使用(§9.9)。

## 3. データ最小化の設計方針(実装済み統制を含む)

| 方針 | 実装状態 |
|---|---|
| calculation_trace / legal_trace に PHI を持たせない(ids・日付・版のみ) | ◎ 実装済み — `@yrese/trace` CalculationInputsSummary は型設計で name/text フィールドを持たない(ddc06a1) |
| 同期イベントの PHI は暗号化必須 | ◎ 実装済み — `@yrese/events` phiClassification≠none → encryptionStatus='encrypted' を実行時強制(85bd3aa) |
| 通常ログへの PHI 平文出力禁止 | ○ 方針確定(§9.6)— log redaction 実装は observability 実装時。エラーメッセージへの PHI 混入禁止は WP-2002 でも適用 |
| agmsg・Codex 入力への PHI 持ち込み禁止 | ◎ 運用中(AGT-010、実績あり) |
| 資格確認結果のスナップショット最小化 | ○ 設計方針(公式仕様確認後に項目確定) |
| サポートアクセスの PHI 最小化・監査 | ○ 設計方針(§9.2) |
| fixtures / テストデータは合成のみ | ◎ 運用中(fixture_policy は WP-0012 で正式化) |

## 4. 権利対応・ガバナンス

- 開示・訂正・利用停止請求への対応: 薬局(管理者)経由の運用フロー(Phase 1 で手順化)
- 委託関係: 薬局=個人情報取扱事業者(管理者)、当社=受託者。データ処理契約で明確化(§9.8)
- 越境移転: なし(AWS東京リージョン固定)
- 保存期間・廃棄: data_governance_policy(WP-0010)で法令保存義務と削除権のバランスを定義。監査ログは一般データと別権限(§9.5)

## 5. リスク評価サマリー

| リスク | 影響 | 発生可能性 | 対応 |
|---|---|---|---|
| ログ経由の PHI 漏えい | 重大 | 中(実装ミス起因) | 型設計による排除 + redaction + レビューゲート(実装済み部分あり) |
| テナント越え開示 | 重大 | 低 | SEC-006 + isolation test 必須化 |
| Edge 盗難による大量漏えい | 重大 | 低〜中 | SEC-005(暗号化・失効) |
| 過剰保持(資格確認結果等) | 中 | 中 | スナップショット最小化 + 保存期間管理【要確認】 |
| 訓練・デモでの実データ使用 | 重大 | 低 | 合成データ強制(§9.9)+ fixtures PHI scan(CI 追加予定) |
