## 冒頭宣言 — 本書の位置づけ(必読)

- 本書の情報源はすべて **Priority C(ベンダー公開ページ・ユーザー提供調査)** である。
- **用途は「薬局業務上の必須要件の抽出」の補助のみ**。実装根拠(evidence_id の源泉)には使用できない。
- **各社実装の模倣を禁止する**。各社の内部算定エンジン仕様(算定ロジック・ルール表・DSL・計算順序・丸め処理・例外処理・テストケース)は非公開であり、模倣の対象自体が存在しない。
- **fable5 決定(2026-07-09)**: 本システムの算定エンジンは、調剤報酬点数表・記録条件仕様・標準仕様・基本マスター・電子レセプト作成手引き(いずれも Priority A)に準拠した**根拠追跡型・決定論的ルールエンジン**として独自設計する。LLM/AI に算定判断をさせない(補助・候補提示・説明文生成のみ可)。

## 1. ベンダー別の公開機能と反映ポイント

### 1.1 MEDIXS(クラウドレセコン)

公開されている特徴: クラウドながらオンプレミス並みの軽快さ、シンプル操作、リモート入力、オフライン利用、窓口業務・レセプト請求・処方集計・日次処理。「クラウドでもサクサク軽快」「オフラインでも安心」「毎月のレセプト請求も3STEP」。薬歴・レセコン・かかりつけ支援・在庫管理を統合したクラウドソリューションとして、複数プロダクト連携で薬局業務を最適化する思想。

**反映ポイント**: レセコン単体ではなく、電子薬歴・在庫・患者接点・経営分析へ拡張可能なモジュール設計(既存の bounded_contexts 設計と整合)。「クラウドでも軽快」「オフラインでも安心」「請求が少ステップ」は UX/SLO 設計(UIX-003/005)にそのまま反映する。

### 1.2 EMシステムズ MAPs for PHARMACY DX / Recepty NEXT

公開されている特徴: レセコン・電子薬歴の同時起動、2次元シンボル・電子処方箋を**会計前に電子薬歴側で確認する密連携**、見やすく使いやすいUI、マウス・キーボード入力、タブ・子画面、各種アプリ連携。患者フォローアップリスト、薬歴作成時間短縮、チェーン店業務・本部システム、在庫管理、処方箋スキャニング、**複数レセコンメーカー混在時のデータ集計**。Recepty NEXT はデータセンター同期・**週1回のマスター配信**・バージョンアップによる最新システム利用。

**反映ポイント**: 会計前に電子薬歴・監査側でも確認できる**イベント連携**(Pharmacy Integration API のイベント配信設計)、マスター配信サイクル(MST-001 の配布設計の相場観)、チェーン本部集計・他社混在データ集計(後続フェーズ・データポータビリティ設計 OPS-011 と関連)。

### 1.3 ウィーメックス / PHC PharnesX-EX・MX

公開されている特徴: **レセプト請求前チェック**、**薬剤服用歴管理指導料を算定しているが薬歴未記載の患者リストアップ**、ファンクションキーによるナビゲーション、薬局予定・配信情報・統計情報・長期投薬患者一覧の表示。薬局内サーバとクラウドサーバの**リアルタイム同期による障害・災害時の運用継続**。MX では重複投薬・同種同効薬・配合剤成分重複・OTC相互作用チェック、患者情報の一画面表示、在宅支援。

**反映ポイント**: **「薬学管理料を算定したのに薬歴未記載」を請求前点検で BLOCKER にする**(不正請求防止 — レセコンと電子薬歴の責務境界にまたがる要件。pre_claim_check_policy / 電子薬歴連携API 契約に反映)。ローカル+クラウドのリアルタイム同期による業務継続は既存 Edge Node 設計(ARC-001/002)と同方向。

### 1.4 モイネット Pharmy Connect

公開されている特徴: 電子薬歴、処方箋QRコード読み込み、帳票、請求業務、NSIPS連携、電子処方箋受付・調剤結果登録、オンライン資格確認、**処方箋OCR**、**長期収載品の選定療養額から消費税まで自動計算**、オンラインアップデート、自動在庫管理。1way入力画面、画面切替の速さ、軽快な操作性、電子処方箋取込時点で薬歴端末にも処方内容と監査結果を自動表示。

**反映ポイント**: 選定療養は**保険分・選定療養分・消費税・患者負担を分離して計算する別建て構造**が必要(selected_medical_care_policy WP-0028)。OCR受け口・1way入力・取込時の薬歴側自動表示(イベント連携)を要件化。

### 1.5 ユニケ P-CUBE n

公開されている特徴: 処方入力、会計・施設請求、オンライン資格確認、**電子調剤録**、QRコード読み取り、**OCRデータ読み取り**、薬袋発行、在庫、電子薬歴、処方監査の一体型。オン資で取得した薬剤情報・特定健診情報の活用、安全管理ガイドライン準拠の調剤録電子化。頭書き情報と薬歴入力の連携、1画面構成、処方薬ごとの指導ガイダンス、LINE服薬フォロー。

**反映ポイント**: 会計・施設請求/電子調剤録/薬袋/QR・OCR/オン資の基本線は MVP で押さえる。服薬フォロー・AI薬歴は後続フェーズ。

### 1.6 ZOO GENNAI just

公開されている特徴: 処方入力・電子薬歴・在庫管理一体型。大きく見やすい処方入力画面、**患者の薬歴情報・引継ぎ事項・現在庫数の同一画面表示**、リアルタイム入力チェック、服薬指導の流れに沿った薬歴ナビモード、処方入力とリアルタイム連動する在庫管理、オンライン更新、QR読み取り、電子調剤録、NSIPS連携。

**反映ポイント**: **処方入力時に「現在庫」「患者情報」「警告」を同一画面で見せる**設計は UX と医療安全の両面で有効 — fast_input_interaction_policy(WP-0032)と在庫連携口(現在庫表示)の要件根拠。

### 1.7 調剤くんV8

公開されている特徴: AI薬歴作成支援、本部入力システム、オンライン服薬指導連携、入力補助、他店舗・本部閲覧、自動アップデート、訪問調剤、ダッシュボード。**初心者にはマニュアルなしでも感覚的に操作できる画面、経験者には必要情報を1画面表示しショートカットキーで高速操作**。電子調剤録、二次元バーコード、画像スキャナー保存、クラウドバックアップ、サブサーバー機能。

**反映ポイント**: **二重UX(初心者=ガイド付き入力/熟練者=ショートカット高速入力)の両立**を体験品質SSOTへ追加(UIX-002/004 改版事項)。サブサーバー・クラウドバックアップの BCP 思想は Edge Node 設計と同方向。

## 2. 横断的な結論

1. **MVPで外してはいけないもの**は「正確な算定・請求」だけではない: 処方入力の速さ / 請求前点検 / オン資・電子処方箋・QR/OCR入力 / 電子薬歴・監査連携 / 在庫連携(最低限は現在庫表示) / 障害時継続 / マスター更新 / 直感的UI。
2. 各社の内部算定エンジンは非公開。公開されているのは外形機能(処方入力・加算可否チェック・請求前チェック・選定療養自動計算・オンライン請求データ出力)のみ。→ 独自設計の正当性(冒頭宣言)。
3. 算定エンジンは「UIの裏の計算関数」ではなく**薬局業務全体の安全制御装置**として設計する(候補抽出と確定算定の分離・薬歴記載との整合・請求前点検との連動)。
4. 体験品質の競争最低線(v0.1.7 §8 と整合、追加分を強調): 処方入力が速い / 画面切替が速い / マニュアルなしで基本操作 / **熟練者はキーボードだけで高速入力** / **初心者はガイド表示で迷わない** / オフライン業務継続 / 請求前に危険なミスを止める / 算定根拠が説明できる / **電子薬歴・監査・在庫と自然につながる** / **自動更新しても請求事故を起こさない**。
