## 業界の構造問題(ユーザー提示・原文趣旨)

1. **NSIPSという古い規格が支配し、業界全体の進歩が止まっている**
2. **品質の低いレセコンがシェアを握り、業界全体の進歩が遅い**
3. **システムの安定性が低く、24時間稼働するシステムになっていない**
4. **システム間の連携基盤が弱いので発展しない**

## 対抗コンセプト: 4本柱

### 柱1 — Post-NSIPS: 接続を人質に取らせない

NSIPS を「敵として排除」するのではなく「周縁化」する。

- **正**: Pharmacy Integration API(OpenAPI 3.1 / OAuth2 CC / mTLS / 署名付きWebhook / イベント駆動 / Idempotency)を薬局内外連携の第一級市民とする。仕様・SDK・sandbox・contract test kit を公開し、パートナーは会員手続きも許諾交渉もなく接続開発を始められる。
- **従**: NSIPS は正規許諾取得時の互換 Adapter(単一薬局内連動限定)として提供し、既存機器を持つ薬局の移行を塞がない。無許諾の複製・模倣はしない(法的清潔さ自体が競争力)。
- **効果**: 「NSIPSにしか繋がらない機器・システム」というロックインの再生産を止める。接続仕様の公開性で機器・システムベンダーの開発コストを下げ、エコシステムの重心を移す。

### 柱2 — 証明可能な品質: 品質をマーケティング語から検証可能な事実へ

低品質シェアへの対抗は「品質が高いと言う」ことではなく「品質を第三者が検証できる」ことである。

- **evidence_id / calculation_trace**: すべての算定・請求・帳票が公式根拠に遡れる。「なぜこの点数か」を薬剤師・事務・監査が説明できる。
- **fail-closed 原則**: 根拠のない計算・未知の状態・未確認の外部結果は「動く」のではなく「止まる」(isClaimable allow-list 化、空ルール→BLOCKED、PHI未暗号化拒否 — すべて実装済み・実行時強制)。
- **公開される品質**: 主要データモデル SSOT・conformance test 結果・SLO 実績を開発者ポータルで公開する。golden test は公式資料の evidence と1:1対応。
- **効果**: 「品質の低いレセコンでも売れる」市場の情報非対称性を崩す。比較可能になった瞬間、検証可能な品質が参入障壁になる。

### 柱3 — 24/365 を設計目標にする: 「夜は止まるもの」を断つ

- **Cloud Core**: zero-downtime 更新(Blue/Green + canary + auto rollback)、expand-migrate-contract、PITR/DR。「メンテナンスのため停止します」を原則排除する。
- **Pharmacy Edge Node**: クラウド障害・回線障害・外部公的システム障害のいずれでも LOCAL_ONLY で受付・仮算定・仮帳票・監査ログが継続する。「クラウドが落ちたら薬局が止まる」構造を設計段階から持たない。
- **正直な継続性**: 24時間「動いているように見せる」のではなく、できること/できないこと/復旧後に必要なことを明示する(§15 の禁止事項・PENDING 系ステータスは本柱の一部)。
- **SLO の公開コミットメント**: 可用性・主要操作レイテンシ・crash-free(OPS-005/UIX-003/005)を内部目標ではなく公開値として運用する。在宅・夜間当番・24時間薬局の実需に応える。

### 柱4 — Integration Hub を中核に: レセコンをハブにする

連携は「オプション機能」ではなくプロダクトの中核である。

- Integration Hub(partner registry / scope・consent / event catalog / Webhook / Outbox・Inbox / 監査 / sandbox / SDK / versioning・deprecation policy)を第一級のドメインとして実装する。
- 電子薬歴・監査機器・在庫・POS・お薬手帳が「接続を申請してから数日で sandbox 接続できる」開発者体験を目標にする。
- 外部ベンダーへの直接DBアクセス・undocumented API・パートナー別の場当たり仕様を禁止する(v0.1.8 §0.0.4.11)。
- **効果**: 連携基盤の弱さで発展が止まる構造を、イベント駆動のハブで反転させる。接続先が増えるほど本プロダクトの価値が増える。

## 実行原則(優先順位判断への反映)

1. 4本柱に資するWPは、同リスク・同コストの他WPより優先する。
2. 柱と矛盾する実装(ロックイン的仕様・fail-open・停止前提メンテ・非公開の場当たりAPI)は BLOCKED とする。
3. 柱は法令適合性・医療安全・請求事故防止の**上には立たない**(v0.1.8 の停止条件が常に優先)。「開かれている」「止まらない」は、正確で安全であることの上に成立する。
4. ベンチマーク(PRD-004/005)は模倣リストではなく、「業界の現在地」の記録として使う。差別化は本書の4本柱で行う。

## 戦略具体化(2026-07-09 追補)

- プロダクト呼称: **Fable-Rx**(ユーザー命名)。
- ワンライナー: **「NSIPSを境界に追放し、イベントログを心臓に据え、品質を公開数字で証明し、APIで生態系を作る、止まらないレセコン」**
- 具体化の正本: 構築プロンプト v0.2.0(docs/spec/construction_prompt_v0.2.0.md)および WP-0041〜0048 の SSOT パック群 — NSIPS隔離アーキテクチャ / FHIR方針(PRD-007: 電子処方箋対応 ≠ JP Core準拠の分離)/ 品質公開KPI(返戻率)/ イベントソーシング+夜間バッチ廃止 / always-on 24/365 / API-first dogfooding+PH-OS参照連携+OSS公開 / WORM監査・テナント分離。
- 追加原則: **夜間バッチの廃止**(締め=ロック操作でありバッチ時間ではない)、**返戻率の公開**(匿名化・同意前提)、**local-first**(Edge Node × 端末PWAキュー)、**最初の接続クライアントは PH-OS**。

## やらないこと

- NSIPS 仕様の無許諾複製・模倣(柱1の法的清潔さを守る)
- 品質実績・SLO 実績の非公開化・粉飾
- 「見かけの稼働」のための失敗隠蔽・成功偽装
- パートナー囲い込みのための仕様非公開・解約時データ人質化(OPS-011 出口戦略が正)
