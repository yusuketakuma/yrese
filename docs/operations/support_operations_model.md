## 1. サポート階層

| 階層 | 担当 | 責務 | PHI アクセス |
|---|---|---|---|
| 薬局内管理者 | 薬局側 | 初期切り分け・端末/デバイス一次対応・ユーザー管理 | 自薬局のみ(業務権限) |
| L1 | サポート窓口 | 問い合わせ受付・分類・既知事象案内・操作案内 | 原則なし(画面共有は薬局同意+監査ログ) |
| L2 | 技術サポート | ログ調査(マスク済み)・設定調査・デバイス調査・再現確認 | マスク済みログのみ。PHI 閲覧は薬局同意+break glass |
| L3 | 開発・SRE | 障害対応・修正・データ修復 | 最小権限+全操作監査(SEC-007) |

問い合わせ分類(初期セット): 操作 / デバイス・印刷 / 資格確認・外部接続 / 算定・請求疑義 / 障害 / 移行 / アカウント・権限 / その他

## 2. リモートサポート統制(v0.1.7 §9.2)

- 本人確認: 薬局管理者の事前登録された連絡先による確認+セッション単位の同意記録
- PHI/PII 最小化: サポート画面はマスキングモードを既定とし、解除は薬局同意+理由記録
- サポート操作監査ログ: 全リモート操作を actor_id=サポート担当として SEC-007 の監査ログへ(support access log)
- 算定・請求・法令判断を根拠なしに回答しない: 回答は evidence_id または公式資料参照付きテンプレートに限定。不明は「確認して回答」フローへ

## 3. 月次請求期の強化体制

- 対象期間: 毎月1日〜10日(オンライン請求期限前後)【候補値】
- 増員・優先キュー: 請求カテゴリの問い合わせを最優先(SLA 短縮)
- 変更凍結: 請求期間中の本番リリースは原則凍結(緊急修正は change_control_policy の例外手続き)
- 請求期障害はデフォルトで severity を1段引き上げ(OPS-013)

## 4. エスカレーション・通知

- 重大障害(請求不能・データ破損・PHI漏えい疑い): 検知から30分以内【候補値】に一次連絡(連絡テンプレート: 影響範囲/現在できること・できないこと/次報時刻)
- 既知障害・既知制限は公開ページで常時掲示(隠蔽禁止 — 「障害を軽く見せるために失敗状態を隠す」の禁止)
- リリースノート・メンテナンス通知: 事前通知期間の標準【候補値: 7日前、緊急時除く】
- 法令・マスター・外部仕様変更時の告知: version_watchlist(REG-002)の検知と連動

## 5. Runbook

- 薬局向け Runbook: 障害時ガイド(LOCAL_ONLY 移行手順・復旧手順)、月次請求期ガイド、デバイストラブルガイド
- サポート向け Runbook: 切り分けフローチャート、ログ調査手順(マスク済み)、エスカレーション基準、break glass 手順
- いずれも Phase 1 で整備し、UIX-002 の「マニュアルレス」原則と整合(Runbook は異常時の補完であり、通常操作をRunbook依存にしない)

## 6. 禁止事項(v0.1.7 §9.2)

- サポート担当者が必要以上の患者情報を閲覧すること
- サポート操作を監査ログなしに行うこと
- 問い合わせ対応で請求・算定・法令判断を根拠なしに回答すること
- 障害を軽く見せるために失敗状態を隠すこと
