## 1. 目的

スキーマ変更を、患者データ・請求データを壊さず・監査可能に・24/365 運用(ARC-010/011)と
両立する形で行うための規律を確定する。ツール選定は WP-5002 に委ね、本書は不変の規律のみを定める。

## 2. 前方一方向の原則

- マイグレーションは**前方(up)のみを正**とする。down migration による本番データの破壊的巻き戻しを
  運用手段にしない。
- 誤ったスキーマ変更の是正は、**新しい前方マイグレーションの追加**で表現する
  (silent fix 禁止・履歴の不可視化禁止と同型)。
- dev/test 環境での down 利用は許容するが、本番適用パスに down を含めない。

## 3. 版管理と適用履歴

- スキーマ版はマイグレーションファイルの単調増加列で管理し、適用履歴を DB 内の履歴テーブルに残す
  (適用日時・適用者・対象版)。
- マイグレーションファイルは一度コミットしたら**改変禁止**(イミュータブル)。修正は新ファイルで。
- リポジトリのマイグレーション列と DB の適用履歴は起動時に照合する。**不一致の意味論**は次のとおり
  区別する(一律の起動拒否は §4 の後方互換3段および ARC-010/011 の 24/365・ローリングデプロイ・
  切り戻しと矛盾するため):
  - **DB が前方互換に先行**(アプリがバンドルする列の prefix と DB 適用履歴が一致し、DB 側に
    追加適用済みマイグレーションがある)→ **許容**。後方互換3段の「旧コードは新スキーマで動く」
    が成立している状態であり、旧バージョンへの切り戻し・ローリング中の旧レプリカ起動を妨げない
  - **同一版のチェックサム相違**(適用済み版とファイルの内容が異なる = 改変検知)→ **起動拒否**
  - **アプリが要求する未適用マイグレーションの存在**(アプリの列が DB より先行)→ **起動拒否**
    (適用は §5 の明示運用操作でのみ行う — デプロイ時の暗黙適用はしない)

## 4. 後方互換の3段適用

破壊的でない変更に分解して適用する。標準パターン:

1. **追加**: カラム追加は nullable または DEFAULT 付きで追加(旧コードが動き続ける)
2. **移行**: アプリ改版+backfill(backfill は再実行可能・冪等に書く)
3. **確定**: 全データ移行の検証後、NOT NULL 化・旧カラム廃止を別マイグレーションで

DROP・型変更・意味変更を含む**破壊的変更は独立レビュー必須**(PRC-005 のゲートに従い、
会計・監査・請求系テーブルは opus4.8 レビューを必須とする)。

## 5. 適用の運用規律

- マイグレーション適用は**明示的な運用操作のみ**。時刻トリガーの自動適用・デプロイ時の
  暗黙適用をしない(ARC-011。適用そのものを監査イベントにする)。
- 本番適用前に、本番相当データのコピー環境で dry-run し、所要時間・ロック影響を計測する
  (24/365 原則: 長時間ロックを伴う変更は 3段分解やオンラインスキーマ変更手段を検討)。
- 適用は QUA-003(変更管理)の変更記録と紐づける。

## 6. 停止条件(fail-closed)

- 本番での down migration 実行 → 禁止(是正は前方で)
- 同一版チェックサム相違・未適用マイグレーション要求での業務開始 → 起動拒否(§3 の意味論。前方互換な DB 先行は許容)
- コミット済みマイグレーションの書き換え → GENERATED_CODE_DRIFT_BLOCKED と同型で拒否
- 破壊的変更の単独適用(3段分解なし・独立レビューなし) → CHANGES_REQUESTED
