## 1. 目的と基本姿勢

調剤録・処方箋・会計・監査等の保存期間と削除の方針を定める。
法定保存年限は本書で**断定しない**(BLOCKED_LEGAL_REVIEW — REG-003 の法令整理で確定)。

**基本姿勢(fail-closed): 年限・削除可否が確定するまで、削除しない側に倒す。**
「消してよいか不明なので消さない」を既定とし、削除は根拠(法令+本 SSOT 改版)を得てから実装する。

## 2. 論理削除と物理削除

| 区分 | 方針 |
|---|---|
| 業務上の「取消・無効化」 | 論理状態(CANCELLED 等の明示ステータス+履歴保持)。行の物理 DELETE で表現しない(P-12: 履歴の不可視化禁止) |
| 監査ログ・会計台帳・確定レセプト | **物理削除 API を作らない**(SEC-008 / ARC-007。年限経過後の扱いも法令確定までは保持) |
| 一時データ(セッション・キャッシュ・Edge 投影) | 派生データ(正本から再構築可能 — ARC-006)に限り、技術的な物理削除を許容 |
| dev/test の合成データ | 制約なし(そもそも PHI を含まない — MOD-013) |

## 3. アーカイブ階層への移動

- 稼働系(hot)から保存系(アーカイブ — PLAN-DB-001 §2 の 12)への移動は「削除」ではなく
  「階層の移動」として扱い、移動記録を残す。
- 移動後も法定保存中のデータは検索・提示可能性を維持する(監査・行政対応で参照できない状態を作らない)。
- 階層・媒体の選定(S3 Glacier 等)は候補に留め、確定は SEC-008 §3 と同じ独立ゲートに従う。

## 4. 患者からの削除要求

- 個人情報保護法上の消去請求等への対応手順は、法定保存義務との競合整理(法令レビュー)の完了後に
  手順化する。**それまで個別対応をアドホックに実装しない**(要求の記録と人間判断へのエスカレーションのみ)。
- 手順化の際も、法定保存義務のあるデータ(調剤録・レセプト等)の破壊は行わない前提で設計する
  (競合時の優先は法令整理の結論に従う — 本書で断定しない)。

## 5. 停止条件(fail-closed)

- 年限未確定データの削除実装 → BLOCKED_LEGAL_REVIEW
- 監査・会計・確定レセプトへの物理削除 API → 実装禁止(SEC-008 §5 と同一)
- 論理削除相当を物理 DELETE で実装 → CHANGES_REQUESTED(履歴不可視化)
- アーカイブ移動の記録なし実装 → SSOT_UPDATE_REQUIRED
