# 03 — Existing Receipt Computer & Electronic Medication Record Benchmark (Phase 2)

公開情報(公式製品ページ・公式マニュアル/カタログ・公的資料・導入事例)のみに基づく**抽象度の高い UX 原則**の抽出。
**固有の画面レイアウト・文言・アイコン・画像・ブランド表現は一切複製していない**(§3.3 / §7.1 遵守)。
競合の多数決でデザインを決めず、本システムへの適用可否(ADOPT/ADAPT/REJECT)を理由付きで判断する。

## 調査カバレッジと限界

- 調査カテゴリ5つのうち **4カテゴリを実施**(レセコン中心 / 電子薬歴中心 / 在宅・モバイル / 公的資料・一般UXパターン)。
- **「統合型/クラウド型」カテゴリ1件は自動調査 subagent が本リポジトリ `CLAUDE.md`(Claude レーン INACTIVE)を読んで停止したため未実施**(捏造で補完しない)。必要なら再調査可。
- 公開情報が乏しい製品の画面仕様は推測補完していない。

## 抽出した抽象パターン(→ 本システムへの適用)

### レセコン中心(患者検索・点検・月次締め)
| 抽象パターン(原則) | 判断 | 本システムへの適用 |
| --- | --- | --- |
| 入力項目順を「一次証憑(処方箋・保険証)の読み取り順」に合わせ転記の視線移動を最小化。自由入力よりマスタ選択優先で誤登録減 | ADOPT | 処方入力 SCR-004 設計原則。マスタ選択 UI |
| **例外駆動(exception-first)**: 全件提示でなく「問題のある対象だけ」を種別層別で抽出 | ADOPT | 請求前点検 SCR-019 / RECOVERY_SYNC 要再検証一覧。対象量を桁で削減 |
| 検証を「後段一括点検」から「入力の瞬間の即時フィードバック」へ前倒し。ブロックすべき危険と通すが警告を区別 | ADOPT | InlineValidation。UIX-001 §5 severity と整合(BLOCKER vs WARNING) |
| 高頻度・定型操作は「事前定義した束の一発呼び出し」+ 一次証憑からの自動取込で最短化 | ADAPT | MVP 範囲で段階導入。誤薬誘発サジェストは避ける(P-10) |
| 月次締めを「少数の順序付き明示ステップ」で提示し各前提充足/完了を伝達 | ADOPT | WorkflowStepper(SCR-020)。手計算排除 |
| 状態(エラー/警告/未点検/解消済)を**一貫した視覚的状態言語**で符号化。ただし色を唯一の手掛かりにしない | ADOPT | **本タスク中核の Visual Status Registry の外部裏付け**(§08) |

### 電子薬歴中心(前回比較・アラート・服薬指導)
| 抽象パターン | 判断 | 適用 |
| --- | --- | --- |
| **差分認識**: 前回との変化を人が読み比べず、システムが整列・強調し「変化箇所だけ」に注意誘導。増/減/新規/中止を色だけでなく記号・ラベルで冗長符号化 | ADOPT | PrescriptionChangeIndicator(§08 軸B)。**処方差分と臨床警告を視覚的に分離**(§11.7) |
| アラートは重大度で階層化、一覧は重大度サマリ→詳細ドリルダウンの二段。禁忌等高危険を最上位 | ADOPT | ClinicalAlertSummary / UIX-001 §5 |
| アラートは「例外が存在する時だけ」出す(常時警告はノイズ化)。最高危険度のみ確認強制、低危険は抑制表示 | ADOPT | warning fatigue 対策(H-08)。CRITICAL 希少性維持 |
| 実施(指導)と記録を二重入力にせず、実施操作の副産物として記録生成。文例は選択式 | ADAPT | 服薬指導記録(将来)。二重入力回避 |
| 患者 identity・重要属性を作業画面に常時随伴、属性に応じ表示(用量閾値・警告)を文脈依存で変える | ADOPT | PatientContextBar 横断固定(L-01)/ P-09 |

### 在宅・モバイル(オフライン・同期・タッチ)
| 抽象パターン | 判断 | 適用 |
| --- | --- | --- |
| ローカル未同期/同期済/同期失敗を**明示的な状態**として扱い、復帰時に自動キュー送信。入力はネットワーク非依存 | ADOPT | system-mode / SyncIndicator(§08 軸D)。H-03 対策 |
| 持ち出しデータの範囲・寿命を明示的に区切る(必要最小限+自動失効) | ADAPT | オフラインデータ設計時の受入基準(将来) |
| オンライン/オフライン・同期済/未同期・情報鮮度差を明示し、利用者が現在の信頼度を判断できる | ADOPT | stale/last-updated 表示(S-02)/ P-02 |
| 真実源を端末から分離、端末は一時ビュー/入力口として単一端末障害を無害化 | ADAPT | Cloud Core + Edge のアーキと整合 |

### 公的資料・一般UXパターン(患者安全)
| 抽象パターン | 判断 | 適用 |
| --- | --- | --- |
| 患者特定は単一識別子に依存せず**最低2つの独立識別子**(氏名+生年月日 等)を同時提示し照合 | ADOPT | 患者検索/ヘッダ。既存の同姓同名警告を強化(H-01) |
| 類似候補が同時に存在する場合、相違点(生年月日・ID 差分)を視覚的に際立たせ選択直前に明示 | ADOPT | 既存 `duplicateKanaSet` の発展。差分強調 |
| 不可逆・重大操作の直前に対象患者と実施内容を再提示する確認チェックポイント | ADOPT | DestructiveActionDialog(P-11)。二段階確認 |
| 警告を重要度で階層化、真に危険なもののみハードストップ、軽微は非中断の受動的様式 | ADOPT | UIX-001 §5 / BLOCKER vs WARNING |
| 重大警告を無視して続行する場合は**理由記録+明示操作**を要求し、誰がいつ何を承知で続行したか追跡 | ADOPT | override 理由 + audit(ClinicalAlert 設計 §11.7) |
| 記録に「未確定(下書き)」と「確定」の明確な状態、確定を境に編集権限・監査扱いを切替、確定後は更新履歴 | ADOPT | RecordLifecycle(§08 軸C)/ P-06,P-07,P-12 |
| 確定記録は物理削除せず訂正履歴を残す。代行入力は実操作者と責任者を分離 | ADOPT | amended/superseded + proxy-entered(§08) |
| 熟練度に依存せず安全側に倒れる既定値・ガード付き導線。頻用は短縮しつつ危険操作にはガード | ADOPT | fail-safe default。既存 fail-closed 方針と一致 |

## コピー回避のための判断

- 採用したのは**情報の優先順位・認知負荷低減・差分認識・患者コンテキスト維持・状態遷移伝達・誤操作防止・例外回復**という
  抽象原則のみ。特定製品の画面構成・配色・アイコン・文言は採らない。
- 本システム固有の制約(依存ライト・system-mode ドメインモデル・fail-closed 請求・PHI 非出力)に接地して取捨選択した。
- 最も強い外部裏付けは「**一貫した視覚的状態言語で状態を符号化しつつ色を唯一の手掛かりにしない**」であり、
  これは本タスクの Visual Status Registry(§08)と UIX-001 P-20 を支持する。
